## 핵심 요약

이번 온보딩의 중심 메시지는 Claude Code를 “개발자용 코딩 도구”로 소개하지 않는 것입니다. 직원에게는 회사 폴더 안의 규칙, 예시, 템플릿, 스킬을 읽고 Brand Genome 방식의 결과물을 만드는 AI 작업 환경으로 설명해야 합니다.

교육 목표는 개인별 AI 활용법을 늘리는 것이 아니라, 회사의 상품기획·마케팅·SCM·그로스·글로벌 업무 방식을 하네스 폴더에 축적하고 재사용하는 운영 체계를 이해시키는 것입니다. 직원은 하네스를 임의로 바꾸는 사람이 아니라, 하네스를 사용해 업무를 수행하고 좋은 사례와 실패 사례를 AX 담당자에게 전달하는 역할입니다.

구성은 이론, 팀별 실습, HTML 보고 실습 순서가 적합합니다. 먼저 Claude Code와 하네스의 의미를 이해시키고, 그 다음 조직도에 있는 실제 부서별 업무에 연결한 뒤, 마지막에 HTML 보고를 공통 활용 사례로 다루는 흐름입니다.

| 구분 | 직원에게 전달할 메시지 | 교육에서 보여줄 장면 |
| --- | --- | --- |
| 도입 목적 | AI를 각자 감으로 쓰지 않고 회사 기준 안에서 같은 방식으로 사용한다 | 일반 챗봇과 Claude Code의 차이 |
| 운영 기준 | 좋은 사용법은 개인 팁이 아니라 조직 자산으로 승격한다 | 하네스 폴더와 공식 산출물 예시 |
| 역할 분담 | 대표님·AX·팀 리더·실무자가 각자 다른 책임을 가진다 | 스킬 후보 승인, 실무 피드백, 최종 기준 승인 |
| 첫 활용 사례 | 보고는 모든 팀이 공통으로 하는 업무이므로 첫 실습에 적합하다 | HTML 보고서가 대표님 판단 화면으로 정리되는 구조 |

## 주요 변화·이슈

현재 구성안의 가장 큰 보완점은 교육 내용보다 산출물의 밀도와 표현 방식입니다. 원문 온보딩 강의안은 도입 배경, 역할 분담, 하네스 정의, 팀별 실습, HTML 보고까지 이어지지만, 기존 포털 보고서는 그 내용을 지나치게 축약해 대표님이 보기에는 교육의 운영 가치가 충분히 드러나지 않았습니다.

두 번째 이슈는 `/report`가 아직 회사 공식 스킬로 굳어지지 않았다는 점입니다. HTML 보고 포털은 MVP로 작동하지만, 보고서 생성 기준이 스킬로 고정되어 있지 않으면 작성자마다 질문 흐름, 표 구성, 시각 자료 배치, 최종 narrative 품질이 흔들릴 수 있습니다. 따라서 `/report`는 “HTML을 만들어주는 명령”이 아니라 보고 유형을 구분하고 대표님 판단 화면으로 정리하는 표준 절차로 설계되어야 합니다.

| 역할 | 온보딩에서 강조할 책임 |
| --- | --- |
| 대표님 / C-Level | AI가 따라야 할 브랜드 방향성, 제품 철학, 최종 판단 기준 승인 |
| AX 담당자 | 하네스 폴더, 스킬, 템플릿, GitHub 저장소 관리 |
| 팀 리더 | 팀별 업무 기준 검수, 스킬 후보 승인·수정·반려 |
| 실무자 | Claude Code로 실제 업무 수행, 좋은 프롬프트·실패 사례·반복 절차 제보 |

직원이 하면 안 되는 일도 명확히 분리해야 합니다. 하네스 폴더 구조를 임의로 바꾸거나, 스킬 파일과 템플릿 파일을 직접 수정하거나, AI 결과물을 검토 없이 제출하는 방식은 금지해야 합니다. 개인 프롬프트를 회사 공식 방식처럼 공유하는 것도 피해야 합니다.

HTML 보고 실습은 교육 마지막에 배치하는 편이 좋습니다. 처음부터 HTML 보고를 강조하면 직원들이 Claude Code를 보고서 만드는 도구로만 이해할 수 있습니다. 먼저 회사 기준을 AI 작업 환경에 이식하는 구조를 설명하고, 마지막에 보고를 첫 공통 활용 사례로 보여주는 순서가 더 안전합니다.
